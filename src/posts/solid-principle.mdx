SOLID 是面向对象编程中的五个核心原则的缩写：单一职责原则 (Single Responsibility Principle, SRP)、开放封闭原则 (Open/Closed Principle, OCP)、里氏替换原则 (Liskov Substitution Principle, LSP)、接口隔离原则 (Interface Segregation Principle, ISP)、依赖倒置原则 (Dependency Inversion Principle, DIP)。目标是增强软件的健壮性、可复用性、可维护性和可扩展性。

# 单一职责原则

一个类应该**有且仅有一个能引起它变化的原因**。也就是说，每个类的职责都是单一的。

如果一个类兼有多个职责，会造成更高的耦合度，一个功能的修改可能会影响其他功能，影响软件的稳定性。因此，每个类的职责单一，更有利于实现**高内聚，低耦合**，使得项目更易于维护。

例如：当我们要编写一个 Agent 系统时，如果将调用 API 获取回复的功能 `sendMessage(msg)` 与查找记忆的功能 `searchMemory(query)` 都塞到同一个类 `Agent` 中，当我们需要更新 Chat API 调用格式时，查找记忆的功能也可能会受到影响，用《软件设计的哲学》中的话来讲，这样做增大了所谓的 **Change amplification**，使得系统的复杂度提高了。而如果将两个功能分别放到不同的类 `LLMClient` 与 `MemoryManager` 中，就完成了职责分离，实现了解耦。

:::warning
当然，任何原则都不是绝对的，如果系统功能比较简单，划分过多的类反而会导致大量**浅模块**的出现，这又增加了系统的复杂度。其他几个原则也是如此，需要灵活运用，避免教条主义。
:::

# 开放封闭原则

一个类应该**对扩展开放，对修改封闭**。一个类在设计时应该允许后续通过增加新代码来扩展功能，而无需改动已有的代码。

这个原则的意义很明显，如果在扩展时经常改动已有代码，很容易导致已实现的功能出现各种各样的问题。而遵循该原则可以使得软件具有更好的**可扩展性**，减小**Change amplification**。

例如：当我们实现 `LLMClient` 时，需要支持 OpenAI、Anthropic、Gemini 等多家供应商，不同供应商提供的 API格式不同，一种做法是定义一个供应商枚举，然后在 `sendMessage(msg)` 函数中采用 if 或 switch 语句来实现对不同供应商 API 格式的处理。但这样做会导致以后添加新的供应商时，都要去修改已有的 `sendMessage(msg)` 函数，这显然违反了开放封闭原则。更好的做法是：为每个供应商定义一个继承自 `LLMClient` 的子类，每个子类重写父类的 `sendMessage(msg)` 函数。这样在添加新的供应商时，只需要添加一个新的子类即可，无需修改原来的代码。

# 里氏替换原则

**基类应该能被子类替换**。具体而言，当某处的基类对象被替换为子类对象时，程序的行为不会发生改变，且不会产生错误。

这个原则的目的在于保证基于多态的代码安全、正常地运行。如果违反了这个原则，继承自同一个基类的子类在调用同一个接口时无法表现出统一的行为，该类有关的代码会变得难以编写和维护。

依旧以 `LLMClient` 为例。在基类中，定义了 `sendImage(img)` 虚函数，用于向 LLM 发送图像。然而有些 LLM 不支持图像识别，例如现在的 DeepSeek。那么，如果让 `DeepSeekClient` 继承自 `LLMClient`，在调用该函数时便无法正常完成预期的功能，或者抛出错误。于是，在所有需要发送图像的场景下，都要考虑当前子类是否正确实现了该功能，为调用者带来了额外的 **Cognitive load** 甚至 **Unknown unknowns**。好的做法是：将发送图像的功能从 `LLMClient` 中分离出来，定义一个额外的接口类 `ImageCapable` 用于实现图像相关方法。`DeepSeekClient` 只继承 `LLMClient`，而支持图像识别的模型供应商，例如 `GeminiClient`，则同时继承两个类。（这也是接口隔离原则的体现）

# 接口隔离原则

不应该强迫客户端使用他们不需要的方法。接口应该**小而专一**，避免“胖接口”。

这个原则的意义是减小调用者的 **Cognitive load**，提高代码的**可读性**，**降低耦合度**。

我们现在来设计 Agent 的记忆系统，需要提供增删查改记忆、备份记忆、回滚记忆功能。一种方案是将所有接口都放到  `MemoryManager` 类中，如果按照这种设计，我们将 `MemoryManager` 作为 `Agent` 类的成员时，就要被迫考虑备份记忆、回滚记忆等功能的调用，被胖接口“绑架”。但是如果我们将记忆系统拆分成 `AgentMemory` 和 `MemoryBackupManager` 两个类，`AgentMemory` 作为 `Agent` 的成员提供记忆的增删查改功能，`MemoryBackupManager` 则提供记忆的备份、回滚功能，在系统的其他位置调用，这样就完成了接口隔离，降低了耦合度，并使得代码更加可读。

# 依赖倒置原则

高层模块不应该依赖低层模块，**二者都应依赖于抽象**；抽象不依赖于细节，**细节应依赖于抽象**。

这个定义似乎很抽象。可以这样理解：不要让头子亲自使唤某个具体的手下去干活，而是让头子给出“战略方针”，当头子下令干活时，手下就根据这份“战略方针”去干活。头子就是高层模块，手下就是低层模块，“战略方针”就是抽象。

高层模块应该直接去定义自己所需要的抽象接口，低层模块实现这些接口，而不是高层模块直接调用低层模块。这样就保证了**低层模块的改动不会使高层模块受到影响**，使得整个系统呈现出“三角形”般的**稳定性**：在低层施工不会使高层倒塌。

我们现在要支持多模态记忆。在之前的记忆系统里，如果我们直接在 `AgentMemory` 中存储 `TextMemory`，使用时直接调用 `TextMemory::getText()` `TextMemory::getKeywords()`……在这时你就会发现：我们只想修改低层模块，高层模块却也要跟着变。但如果我们定义一个 `Memory` 抽象类，提供 `getContent()` `getKeywords()` 等接口，`TextMemory` `ImageMemory` `AudioMemory` 都派生自这个类，`AgentMemory` 也调用这个类的接口，那么后续修改某个记忆或添加新的记忆类型时，高层的 `AgentMemory` 就不用跟着变咯，同时也满足了开放封闭原则。

# 各个原则之间的关系

- SRP 是基础，使得类小而专注，为其他原则的落实提供条件。
- OCP 建立在对 SRP 拆分的稳定抽象上，并通过 DIP 将抽象与实现分离。
- LSP 保证了基于抽象的多态安全，使得 OCP 和 DIP 能够放心运用继承和子类型替换。
- ISP 进一步防止抽象膨胀，让接口恰到好处，巩固了 DIP 和 OCP 的灵活性。

# 温馨提示

再次强调：软件设计原则是**指导性思想而非教条**，需要**灵活运用**。

在实际开发中合理使用 SOLID 原则，是构建可读、可复用、可扩展、健壮稳定的代码的重要一环。编写高质量代码，成为人类高质量程序员！
